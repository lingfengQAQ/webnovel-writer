● 分析完了。Trellis 已装好（仓库里 .trellis/ + .claude/，git status
  干净没污染仓库），它的多宿主机制也摸清了。结论先行：它的做法和我们多宿主 spec 
  v3.1 的方向高度一致，但有四个值得抄的实现细节和一个值得警惕的反面教材。

  Trellis 多宿主架构(14 个平台)

  分层:
  - .trellis/ = 平台无关的核心:Python 脚本(业务逻辑)+
  spec/tasks/workspace(数据)。所有平台的壳最终都调同一句 python3 
  ./.trellis/scripts/get_context.py——和我们"所有宿主复用同一套 Node
  脚本"完全同构。
  - 模板分三层:templates/common/(跨平台共享的 skills/commands
  正文)、templates/<platform>/(每平台特有物:Claude 是
  agents/*.md+settings.json,Codex 是
  agents/*.toml+config.toml)、templates/shared-hooks/(hook 脚本共享)。
  - 平台注册表 AI_TOOLS(单一事实源):每个平台一条记录,声明
  templateDirs、configDir、cliFlag,以及一个 templateContext。

  四个值得抄的细节

  1. 模板上下文 + 条件块,而不是每平台一份模板。共享模板里用
  {{CMD_REF:finish-work}}(各平台命令引用语法不同:Claude /trellis:x、Cursor
  /trellis-x、Codex $x)、{{PYTHON_CMD}},以及
  {{#AGENT_CAPABLE}}...{{/AGENT_CAPABLE}}、{{#HAS_HOOKS}}
  条件块——降级路径直接编译进生成物,没有 subagent 的平台生成出来的 skill
  正文就是顺序执行版。这比我们 spec 里"skill
  写降级路径"更进一步:降级不是运行时判断,是安装时定稿。
  2. 能力声明就两个布尔:agentCapable、hasHooks。没有维护大能力矩阵,和我们"不相信
  手写矩阵"原则一致,但它更狠——只声明模板渲染真正需要的两个维度。
  3. AGENTS.md 作为最大公约数,用 <!-- TRELLIS:START/END --> 标记块管理:块内允许
  工具覆写、块外用户内容保留。任何平台哪怕什么都不支持,至少能读 AGENTS.md 找到
  .trellis/ 入口。我们的 spec 没提 AGENTS.md,建议补——它已经是事实标准。
  4. update 机制:安装时写 .template-hashes.json 记录每个模板文件的哈希,trellis 
  update 时哈希没变的文件直接更新、用户改过的文件提示——解决了"框架升级 vs
  用户已手改生成物"的冲突。我们的 npx 安装器 spec 只写了 init,没写
  update,这是个缺口。

  另外一个细节:Windows 的 Python 命令探测(python/python3/py 
  -3)在安装时探测后写死进所有生成物——这正是我们选 Node
  要避免的整类问题,反向印证了 Node 决策。

  一个反面教材

  Trellis 的 subagent 提示词里有大段 Recursion Guard("你已经是被派遣的
  trellis-check,不要再派遣 trellis-check")和 hook
  注入失败的兜底协议("如果没看到注入标记,说明 hook
  没生效,自己去读这些文件")。这是 hook/subagent 跨平台不可靠的直接代价——它选择用
  prompt 打补丁。我们 v3.1 的"hook
  只是自动兜底、核心流程显式调脚本"路线避开了这整类问题,坚持住。

● 有道理——"坑"在圈里其实是指整本书的：挖坑=开新书、弃坑=断更整本、入坑=读者开始
  追，拿来指单条伏笔会撞车，语义全乱。是我想岔了。

  那这个概念（伏笔/悬念/感情线/立旗/爽点铺垫的统一体）换两个候选：

  方案 A：伏笔（编号 伏-031）
  - 操作：埋下 / 推进 / 回收 / 
  放弃——"埋伏笔""回收伏笔"是圈内最高频的原生短语，v6 的命令行参数都叫
  --add-foreshadowing
  - 陈坑 → "悬了太久"预警（如"伏-031 已 12 章未动"）
  - 代价：严格说悬念/感情线不算"伏笔"，是概念外延略微撑大——但作者日常本来就把这
  些混着叫

  方案 B：线（编号 线-031）
  - 操作：开线 / 推进 / 收线 / 
  断线——"主线/支线/暗线/感情线""把这条线收了"也全是原生话
  - 优点：外延天然涵盖感情线、悬念（暗线）；缺点："收线"不如"回收伏笔"那么口语化
  高频

  方案 C：双轨——界面上按类型显示原生叫法（伏笔用埋/回收，感情线用开线/收线），系
  统内部仍是同一张表
  - 优点：每个词都最自然；缺点：作者要面对两套动词，文档变啰嗦

  我推荐 A：识别度最高、零学习成本，"回收伏笔"四个字本身就是网文写作论坛的日常用
  语；外延撑大一点没人会困惑（"这条感情线也登记成伏笔"不会有人看不懂，spec
  内部定义清楚就行）。

  你的语感选哪个？
